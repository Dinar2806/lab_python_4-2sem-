## Теоретическая часть

### Протокол итерации и Итераторы

**Итерируемый объект (Iterable)** — это объект, который может возвращать свои члены по одному. В Python объект считается итерируемым, если у него есть метод `__iter__()`, возвращающий итератор. Это позволяет использовать объект в циклах `for`, функциях `list()`, `sum()` и других конструкциях, ожидающих последовательность данных.

**Итератор (Iterator)** — это объект, который реализует методы `__iter__()` и `__next__()`. Он хранит состояние текущего положения в коллекции. Когда элементы заканчиваются, метод `__next__()` должен выбросить исключение `StopIteration`, сигнализируя о завершении обхода.

### Ленивые вычисления (Lazy Evaluation)

**Ленивые вычисления** — стратегия отложенного выполнения операций до момента, когда результат действительно необходим. В контексте коллекций это означает, что данные не загружаются в память целиком и не копируются при фильтрации. Вместо этого создается цепочка инструкций (генераторов), которые выполняются по ходу получения при итерации.

Преимущества:
1.  **Экономия памяти:** Отсутствие промежуточных списков результатов.
2.  **Производительность:** Мгновенный старт обработки без ожидания полной загрузки данных.
3.  **Масштабируемость:** Возможность работать с потоками данных, объем которых превышает размер оперативной памяти.

### Генераторы и Фабрики итераторов

**Генератор** — специальный вид итератора, создаваемый функцией с ключевым словом `yield`. Генератор автоматически управляет своим состоянием и выбрасывает `StopIteration` при исчерпании данных.

**Фабрика итераторов** — функция, которая при каждом вызове создает новый, независимый итератор. Этот паттерн критически важен для реализации требования **«повторный обход»** в ленивых коллекциях, так как обычные генераторы являются одноразовыми.

### Ключевые особенности реализации

1.  **Поддержка повторного обхода**:
    *   Реализована через проверку типа источника (`callable` vs `iterable`).

    *   Они возвращают новый объект `TaskQueue`, источником которого является функция-генератор.
    *   Проверка условий происходит только в момент итерации (вызова `next()` или цикла `for`).

    *   Фильтры по строкам (`reg_filter_by_id`, `reg_filter_by_payload`) используют приведение к `str()`, чтобы корректно работать с числовыми ID и сложными типами payload.

Обычные генераторы одноразовые. Чтобы пройти по ним дважды, нужно создать новый генератор.

***Ключевая особенность этого механизма в данной работе:***
Это позволяет применять фильтры многократно к одному источнику, под капотом это все равно будет реализовано как повторный возврат функции, возвращающая генератор, и вызывающаяся при следующем вызове __iter__

Данные «протекают» сквозь цепочку фильтров по одной штуке. Это обеспечивает сложность по памяти O(1) относительно количества элементов (не считая исходного источника).

2. *Технические требования*
- **Использование генераторов для ленивых операций**
Где в коде: Ключевое слово yield внутри функций _factory во всех методах фильтрации.
Как это работает:
Функции вида:
```
def _factory():
    for task in TaskQueue(current_source):
        if condition:
            yield task
```
Являются генераторными функциями. Они приостанавливают свое выполнение после каждого yield, отдавая управление обратно вызывающему коду, и возобновляют его при запросе следующего элемента. Это основа ленивой обработки.
- **Отсутствие избыточного хранения данных в памяти**
Где в коде: Структура классов и передача данных через ссылки/замыкания.
Как это работает:
При цепочке q.filter(...).filter(...) не создается копия данных. Новая очередь хранит только ссылку (current_source) на предыдущую очередь и правило фильтрации.
Исходные данные (список задач) хранятся в одном экземпляре. Все операции чтения происходят поверх этого единого источника.
Нет дублирования объектов Task в промежуточных буферах.
- **Корректная обработка StopIteration**
Где в коде: Неявная обработка механизмом Python.
Как это работает:
Вам не нужно писать raise StopIteration вручную.
Когда генератор (_factory) доходит до конца цикла for task in ..., он естественным образом завершается.
Python автоматически выбрасывает исключение StopIteration в этом месте.
Конструкции for, list(), sum() перехватывают это исключение и корректно завершают итерацию.
Благодаря тому, что вы используете стандартный цикл for внутри генераторов, механизм исчерпания потока работает штатно и безопасно.

### Дескрипторы в Python

**Дескриптор** — это объект класса, который определяет, как будут получаться, устанавливаться или удаляться атрибуты другого класса. Дескрипторы позволяют перехватывать доступ к атрибутам, добавляя логику валидации, логирования или вычислений.

В Python различают два типа дескрипторов:
1.  **Data Descriptors**: Реализуют методы `__get__` и `__set__`. Имеют приоритет над словарем экземпляра (`__dict__`). Используются для строгой валидации данных (например, проверка диапазона чисел).
2.  **Non-Data Descriptors**: Реализуют только метод `__get__`. Не имеют приоритета над `__dict__`. Если в экземпляре создать атрибут с таким же именем, он "перекроет" дескриптор. Часто используются для кэширования или предоставления информации только для чтения.

### Инкапсуляция и защита инвариантов

**Инкапсуляция** — механизм сокрытия внутренней реализации объекта от внешнего мира. В контексте данной работы она достигается за счет:
*   Использования приватных атрибутов (с префиксом `_`) для хранения реальных данных.
*   Предоставления публичного доступа только через дескрипторы и свойства (`@property`).

**Инвариант класса** — условие, которое должно быть истинным для любого корректного экземпляра класса в любой момент времени. Например:
*   `priority` всегда находится в диапазоне [1, 10].
*   `status` всегда является допустимым значением перечисления `TaskStatus`.
*   `id` никогда не равен `None`.

Нарушение инварианта приводит к генерации специализированных исключений, что предотвращает перевод объекта в некорректное состояние.

### Property vs Descriptors

*   `@property` используется для создания вычисляемых атрибутов или геттеров/сеттеров для конкретных полей внутри одного класса.
*   **Дескрипторы** более универсальны: один дескриптор может быть использован для валидации множества разных атрибутов в разных классах, обеспечивая переиспользование кода валидации.

### Duck typing и контрактное программирование

**Duck typing** — концепция в программировании, основанная на принципе: "Если это выглядит как утка и крякает как утка, то это утка". В контексте данной работы это означает, что объект считается источником задач, если у него есть метод `get_tasks()`, независимо от его типа и наследования.

**Контрактное программирование** — подход к проектированию, при котором компоненты взаимодействуют на основе четко определенных соглашений (контрактов). В Python контракты описываются с помощью `typing.Protocol`.

### Protocol и runtime_checkable

`Protocol` (из модуля `typing`) позволяет определить интерфейс, которому должен соответствовать объект. Декоратор `@runtime_checkable` дает возможность использовать `isinstance()` для проверки соответствия протоколу во время выполнения.

```python
from typing import Protocol, runtime_checkable
from src.task import Task

@runtime_checkable
class TaskSourceProtocol(Protocol):
    def get_tasks(self) -> list[Task]:
        """Получить список задач из источника"""
        ...
```
